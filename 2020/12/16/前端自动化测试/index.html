前端自动化测试

前言

什么是前端自动化测试?

前端自动化测试就是通过编写程序可读的测试用例，通过运行程序自动检测你的css改动是否导致页面错位、js改动是否导致功能不正常。减少回归测试的成本，以及人工测试的遗漏。

为什么要做前端自动化测试？

同上，

  1. 迭代时不用担心影响其他模块
  2. 提高代码质量
  3. 一定的文档作用（可以通过测试用例的描述了解模块的逻辑）

其中最重要的就是第一条：减少 bug 出现的概率，尤其是回归测试的 bug 数。

应用场景

一、给业务主流程编写测试用例
当我们改动代码的时候，这些主流程的测试用例会重新跑一遍，如果影响了主流程，会及时给出错误提示。

二、给公共组件和公共函数编写测试用例
组件或函数有扩展和优化，之前的用例依然能够跑通，就有极大的可能不会影响到之前调用的地方

测试的几个概念

一、黑盒测试
功能测试，是通过测试来检测每个功能是否都能正常使用。

二、白盒测试
代码逻辑测试，基于代码的测试。

三、单元测试
对程序中最小可测试单元进行的测试，例如测试一个函数、一个模块、一个组件…

四、集成测试
将互相之间有包含关系的单元进行组合测试，例如测试包含Column和Row等其他组件的Table组件

五、TDD
测试驱动开发 (Test-Driven Development)。测试流程：

  1. 编写测试用例
  2. 运行测试，测试用例无法通过测试
  3. 编写代码，使测试用例通过测试
  4. 优化代码，完成开发
  5. 重复上述步骤

六、BDD
行为驱动开发 (Behavior Driven Development)。先写代码，然后基于用户行为，编写测试用例。

一些实际应用的结论

集成测试 + BDD：
和组件里的数据耦合性很小，能够很好的保证项目质量，保证用户在使用的时候没有任何问题
但是代码测试覆盖率相对较低

单元测试 + TDD：
和组件里的数据耦合性太强，只保证单个文件测试通过，无法保证整体项目质量

测试业务时： 集成测试 + BDD
测试单一函数时：单元测试 + TDD
实际开发中可以结合使用。

另外，测试覆盖率并不能代表一切，要结合实际情况作出选择。

下面介绍下常用的 jest 断言和测试方法，无聊但有用：

一、jest常用匹配器

1. toBe
相当于js中的 === ， 匹配类型和内容，因此无法校验引用类型

2. toEqual
只匹配内容，不匹配类型，可以判断对象内容是否相等

3. toBeNull
判断是否是 null

4. toBeUndefined
判断是否是 undefined

5. toBeDefined
判断是否定义过

6. toBeFalsy
判断是否是 false

7. toBeTruthy
判断是否是 true

8. not
取反

9. toBeGreaterThan
数字 - 是否大于

10. toBeLessThan
数字 - 是否小于

11. toBeGreaterThanOrEqual
数字 - 是否大于等于

12. toBeLessThanOrEqual
数字 - 是否小于等于

13. toBeCloseTo
数字 - 浮点数约等于

14. toMatch
String - 判断是否包含该字符串

15. toContain
Array, Set - 判断数组中是否包含项

16. toThrow
判断是否抛出异常

二、jest中的钩子函数和作用域

四个核心钩子

1. beforeAll

在所有测试用例执行之前执行一次

2. afterAll

当所有测试用例都执行结束后执行

3. beforeEach

每个测试用例执行之前执行一遍

4. afterEach

每个测试用例执行之后执行一遍

jest中的mock

jest.fn()
  1. 捕获函数的调用和返回结果，以及this和调用顺序
  2. 它可以让我们自由的设置返回结果
  3. 改变函数的内部实现
jest.fn().mock

jest.fn().mock 返回一个对象，包含四个参数

  • calls 被调用的次数及每次调用时传递进去的参数
  • instances 被调用的次数及每次调用时this的指向
  • invocationCallOrder 函数执行的顺序，调用三次即 [1, 2, 3]
  • result 被调用的次数及每次调用后返回的结果
